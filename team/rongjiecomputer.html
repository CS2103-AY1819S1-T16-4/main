PROJECT: ModulePlanner

Overview

This is a project portfolio about a team project called Module Planner when taking CS2103T, Software Engineering.

Module Planner is a desktop module planning application to help NUS Computer Science students figuring out what modules in order to graduate. The user interacts with it using a Command Line Interface (CLI), and it has a Graphical User Interface (GUI) created with JavaFX.

The main features are the adding and deleting of modules, as well as the suggesting of new modules that the user can take based on their academic requirements.

Setting user profile : setup

Initialises your profile with your major and optionally your focus areas so that Module Planner can give you module suggestions that are more relevant to you.
Format: setup m/MAJOR [f/FOCUS_AREA]…​

  • The major and focus areas should be spelt out in full.

  • The major and focus areas are case insensitive. e.g. Computer Science is the same as cOmpUter scienCe.

  • This feature currently only supports majors and focus areas listed below. Support for other majors and focus areas will come beyond v2.0.

    • Major:

      • Computer Science

      • Computer Engineering

    • Focus area:

      • Algorithms and Theory

      • Artificial Intelligence

      • Software Engineering

  • Default user profile is set to Computer Science major with no focus area.

Examples:

  • setup m/Computer Engineering

  • setup m/Computer Science f/Software Engineering

Finding information on a module : find

Retrieves the information of the specified module.
Format: find c/MODULE_CODE

Example:

  • find c/CS1010

You should see the module information for CS1010 in the Multipurpose Panel.

Information about modules and academic majors supports almost all the features in Module Planner.

Simplifications

The real information about NUS modules are very complicated. Here we list down some simplifications we made in order to create this proof of concept of our application.

Prerequisites and preclusion

IVLE provides the prerequisites and preclusions of a module in normal English sentence (e.g. "CS1010 or its equivalents" and "Departmental approval"). Extracting the real module prerequisite and preclusion relationship would be infeasible due to the potential long list of exceptions we need to handle. To simplify this, we use regular expression to extract the module codes.

Here is how prerequisite and preclusion checking is implemented:

  • If user wants to take Module A, all prerequisites of Module A must already have been taken.

  • If Module B’s preclusions has at least one module that user has already taken, user cannot take Module B.

This gives us a reasonable approximation of prerequisite and preclusion relationship for most modules.

Corequisites

Corequisite is used to describe which modules must be taken together. We choose to ignore this entirely as it will complicate the implementation of add and delete commands significantly.

Old modules that have been phased out

NUS sometimes creates new modules to replace outdated modules every academic year. Information about outdated modules are not given by IVLE, but other modules might still list these module codes in their prerequisites and preclusion. We assume that all module codes referenced in our processed data must have a ModuleInfo object. To do that, all references of module codes that do not have its own module information entry are removed.

Design Consideration

Aspect: Representation of academic major and focus area

  • Alternative 1 (current choice): Use enum to represent academic major and focus area.

    • Pros: Stronger type checking.

    • Cons: Need to explicitly list down all the possible values of enums (there are more than hundreds of focus areas in all NUS academic majors). Special JSON serialization and deserialization is needed for enum types.

  • Alternative 2: Use String to represent academic major and focus area.

    • Pros: No need to convert academic major and focus area inputted by user into another type.

    • Cons: If academic major/focus area is passed into a function as a parameter, the function cannot assume that the it is a valid academic major/focus area. This leads to a lot of redundant checking.

Aspect: Storing of module and academic major information

To avoid having to download the module and academic major information from the Internet in first start up, we package the data directly into final .jar file as a resource file.

  • Alternative 1 (current choice): Store data in as a JSON resource file and deserialize data into Java types at runtime.

    • Pros: JSON format is easy to learn and can be manipulated easily with other (interpreted) programming languages.

    • Cons: JSON format can only hold simple primitive types such as String, number, array and hash table. Need to design a way to flatten the module dependency graph into something that can be represented in JSON and reconstruct the graph after deserialization.

  • Alternative 2: Generate Java code during build time. The data will be directly stored as a static constant Java object.

    • Pros: Better performance in runtime as the deserialization is no longer needed. Less moving parts when testing the data itself.

    • Cons: More complex build procedures. Teammates will need to install extra tools to build the project.

Aspect: Class representation of module related information

  • Alternative 1 (current choice): Use ModuleInfo class to represent module information that is transparent to user’s academic major and ModuleDescription class to represent module information that is specific to the requirements of an academic major.

    • Pros: Clear separation of concern. Commands that does not depend on UserProfile only needs to interact with ModuleInfo and not affected by changes to ModuleDescription.

    • Cons: Commands that need module information specific to an academic major (e.g. suggest and status) still need to depend on ModuleInfo.

  • Alternative 2: Use one class to hold all types of module information.

    • Pros: Less classes to reason about.

    • Cons: Commands that use ModuleInfo but do not uses UserProfile directly will still be affected by changes in UserProfile. The JSON form of the data will be more complex.

Design Considerations

Aspect: Argument leniency

  • Alternative 1: Entirely reject all focus areas even if only one of them is invalid. Focus areas must have the exact same letter casing as listed in User Guide.

    • Pros: Implementation would be simpler.

    • Cons: Not user friendly. It is too easy for user to input focus areas in the wrong letter casing, causing many retries.

  • Alternative 2 (current choice): Accept the valid focus areas even if one of them is invalid. Focus areas inputted is case-insensitive.

    • Pros: More user friendly. User is able to update the user profile and gets positive result even if there are some minor syntax errors.

    • Cons: Need to implement filtering of valid focus areas from inputted focus areas. As f/Computer Science and f/cOmpUter SciEnce are now both valid inputs and can be inputted at the same time, we also need to filter out duplicates.

Aspect: Error handling

  • Alternative 1: Execute syntax format checkings one by one and immediately throw command exception in first failure.

    • Pros: Implementation would be simpler. Less possible of negative test cases to test.

    • Cons: Not user friendly. User only sees one syntax error at one time. If there are many syntax errors in user’s first try, it will take several retries for user to figure all the syntax errors.

  • Alternative 2 (current choice): Always execute all syntax format checkings. If there are more than one error, concatenate their error message before throwing command exception.

    • Pros: More user friendly. User will be able to see all syntax errors in first try and attempt to fix them.

    • Cons: Need to create many negative test cases to test each error individually and all together. More complicated to test the expected error messages.
